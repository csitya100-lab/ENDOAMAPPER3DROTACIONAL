Quero implementar no projeto ENDOMAPPER-3D-ROTACIONAL um fluxo de compartilhamento do modelo 3D pensado para médicos que “só querem receber e clicar”, com foco em compatibilidade iOS (Safari) e Android/desktop.

Objetivo de UX (não mude isso):

O médico recebe um link único.

Ao clicar, abre diretamente uma página web com o modelo 3D já carregado, com rotação/zoom por gesto, sem precisar baixar arquivo nem descompactar nada.

Opcionalmente, haverá um link secundário discreto para baixar o arquivo .glb para uso em outro software.

O que preciso que você implemente:

Rota dedicada para visualização 3D

Criar uma rota de front-end, por exemplo:

/3d/:caseId (ou similar, use o padrão de rotas já existente no projeto).

Essa página deve:

Carregar o modelo 3D GLB/GLTF correspondente ao caseId (pode, por enquanto, usar um GLB fixo de exemplo se ainda não houver backend de casos).

Renderizar o modelo em um viewer 3D com rotação e zoom (pode ser Three.js / React Three Fiber ou <model-viewer>, escolha o mais simples e compatível com iOS/Safari).

Não depender de bibliotecas externas pesadas fora do que o projeto já usa.

Layout da página /3d/:caseId
Implementar layout bem simples, focado em usabilidade para médico:

Topo:

Um título curto, ex.: Modelo 3D – Paciente / Caso {{caseId}}.

Uma frase curta de instrução, ex.:

“Arraste para girar. Use rolagem ou pinça para zoom.”

Centro:

Viewer 3D ocupando quase toda a área visível, com fundo neutro (ex.: cinza claro ou branco) e sem menus complexos.

Controles padrão de interação:

Arrastar = rotacionar.

Scroll / pinça = zoom.

Rodapé:

Um link/botão discreto com o texto:

“Opcional: baixar modelo 3D (.glb) para uso em outro software.”

Esse link deve disparar o download do arquivo GLB correspondente ao caso (ou do GLB fixo, na versão inicial).

Formato de arquivo e compatibilidade

Padronizar o formato do modelo em GLB/GLTF.

Garantir que o viewer funcione em:

Safari (iOS – iPhone/iPad).

Chrome/Edge/Firefox em desktop.

Se necessário, ajustar configurações do viewer (iluminação, câmera inicial, limites de zoom) para o modelo pélvico já existente no projeto.

Integração com o fluxo de exportação/compartilhamento

Criar uma função/endpoint que gere um link único por caso (pode ser estático por enquanto, ex.: /3d/123), que será usado no laudo/e-mail.

Deixar o código preparado para, no futuro, receber caseId real vindo de outro app (não precisa implementar o backend completo agora, mas deixe claro onde esse ID será usado).

Textos que já devem ficar prontos na interface (para médicos)

Na página /3d/:caseId:

Instrução curta:

“Arraste para girar. Use rolagem ou pinça para zoom.”

Link secundário para download:

“Opcional: baixar modelo 3D (.glb) para uso em outro software.”

Cuidados importantes

Não remover nem quebrar o que já está funcionando hoje no viewer 3D atual do projeto.

Comentar claramente no código os novos componentes/rotas criados (ex.: // Página de visualização 3D para compartilhamento com médicos).

Organizar o código de forma simples para futura integração com backend (caseId real).

No final, me mostre:

Quais arquivos foram criados/alterados.

Qual é a URL/rota final para um caso de exemplo (por ex.: /3d/demo ou /3d/123).

Um exemplo de link que eu possa usar no laudo/e-mail (ex.: https://<meu-dominio>/3d/123).
